# mitmproxy

Custom mitmproxy image: official `mitmproxy/mitmproxy:latest` + `httpx`. We
need httpx for the addon to call rules-svc; the base image's Python is
minimal and doesn't ship it.

## What this service does

* Runs `mitmdump --mode transparent` inside the **wg container's network
  namespace** (`network_mode: service:wg` in compose).
* Loads `services/rules-svc/addons/mitm_capture.py` (mounted read-only at
  `/addons/`) — this is where the actual policy logic lives.
* Listens on `:8080` inside the wg netns. nftables DNATs HTTP/HTTPS from
  devices in the `mitm_clients` set to `127.0.0.1:8080`, where mitmproxy
  uses `SO_ORIGINAL_DST` (via conntrack) to discover the real upstream and
  forward there.

## Image construction

```Dockerfile
FROM mitmproxy/mitmproxy:latest
USER root
RUN pip install --no-cache-dir --break-system-packages httpx
# Do NOT switch USER back to mitmproxy — the base entrypoint runs as root
# and drops to the mitmproxy user via su-exec. If we run it as mitmproxy
# already, su-exec fails with "operation not permitted".
```

## How the addon works (high level — see `addons/mitm_capture.py`)

For every decrypted request:

1. Extract `client_ip` from `flow.client_conn.peername` (this is the kid's
   real `10.13.13.x` because we're in wg's netns and there's no SNAT in
   the path).
2. `POST /api/decision` to rules-svc with `{client_ip, host, path, query}`.
3. On `block`, short-circuit by setting `flow.response` to a 403 with the
   inlined "You shall not pass" HTML.
4. Fire-and-forget `POST /api/events` with the full record.

For TLS connections we can't decrypt (CA not installed on the device or
pinned app), the `tls_failed_client` hook fires and we log SNI-only events
(filtered by the noise list — googleapis / gstatic / firebase / apple etc.
are dropped so they don't drown the activity log).

## Request classification (the `kind` field)

The addon's `_classify(req)` function decides `page | iframe | xhr | ws |
asset | unknown` from request headers. Order of confidence:

1. WebSocket upgrade (`Upgrade: websocket`) → `ws`
2. `Sec-Fetch-Dest` (browser's own declaration) — `document` → page,
   `iframe/frame/nested-document` → iframe, `image/script/style/font/...`
   → asset, `empty` → xhr.
3. Request `Content-Type` (`application/json`, `application/graphql`,
   `application/x-www-form-urlencoded`, etc.) → xhr
4. `Accept` header — first preference `text/html` → page, `application/json`
   → xhr.
5. Non-GET method → xhr (POSTs aren't navigations).
6. URL extension fallback (`.css/.js/.png/...`) → asset.
7. Default → page (only when every signal is silent).

The activity log default-hides everything except `page` — see rules-svc's
`_activity_events()`.

## Operational gotchas

* **FD exhaustion is real.** mitmproxy holds open connections to upstreams
  for the duration of HTTP/2 streams. Default file-descriptor limits are
  ~1024 — burst traffic from a single device (e.g. opening Reddit) blows
  through that. Compose sets `ulimits.nofile.{soft,hard}: 65536`. Symptom
  is `OSError: [Errno 24] Too many open files` in mitm logs and connections
  silently dying.

* **QUIC bypass.** mitmproxy only does TCP. Any kid using HTTP/3 over QUIC
  bypasses inspection. nftables drops UDP/443 for `mitm_clients` so
  browsers fall back to TCP within ~1s.

* **`.format()` on HTML.** Don't. CSS uses literal `{` / `}` and `.format()`
  will throw `KeyError: 'font-family'`. When the addon throws during the
  request handler, mitmproxy **silently** passes the request through to
  upstream and `_post_event` never runs — block decisions effectively
  disappear. The block page is now a static `bytes` blob to make that
  impossible. If you must template, double every CSS brace.

* **Don't switch USER in the Dockerfile** after `USER root`. The base
  image's entrypoint expects to be root so it can `su-exec mitmproxy`.

* **CA expiry: 3 years from generation.** Generated by `scripts/gen-ca.sh`
  on first `./gdlf init`. Rotating means reinstalling on every device.
  Calendar reminder material.

* **`flow.client_conn.peername` is the kid's IP, not the docker bridge IP.**
  This only works because mitm shares wg's netns. If you ever move mitm
  back onto the gdlf bridge, every event will look like it came from
  `10.42.0.2` and per-kid attribution breaks.

## Where to make changes

Almost all behavior lives in `services/rules-svc/addons/mitm_capture.py`,
not in this directory. This directory only contains the Dockerfile that
adds httpx. After changing the addon, mitmproxy hot-reloads it within a
few seconds (look for `Loading script /addons/mitm_capture.py` in the
logs); no rebuild needed unless you change dependencies.
